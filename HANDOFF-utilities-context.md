# Handoff: utility agent context (manifests → markdown) + llms.txt index

Untracked handoff plan (2026-07-15, written by Fable). Branch: `components`. Builds on `build:context` (component context, `scripts/build-agent-context.ts`) and the utility-manifest system (20 manifests, 117 families, `src/scripts/utilities/`).

**Mark reviews and commits himself — do NOT run `git commit`. No back-compat: restructure llms.txt cleanly, don't dual-publish old and new shapes.**

## Goal

Generate agent-readable markdown for the utility-class system from the utility manifests, shipped exactly like component context: tracked in `docs/static/zebkit/context/`, drift-checked by `check:context`, copied to `dist/cli/context/` by `build:cli`, delivered to consumer repos by `zebkit init`/`zebkit pull`. Restructure `llms.txt` as a true index (the llmstxt.org shape: links + one-liners) and add `llms-full.txt` for whoever wants one giant read.

## Layout decision (made — rationale below)

**Domain files + an index, not one giant utilities file and not 20 per-manifest files.**

| File | Manifests |
|---|---|
| `utilities-layout.md` | flex, grid, layout, object, overflow, visibility |
| `utilities-spacing.md` | margin, padding, spacing (sizes + position), semantic spacing (page/section) |
| `utilities-typography.md` | typography, text, prose |
| `utilities-borders.md` | border |
| `utilities-effects.md` | elevation, opacity, z-index, transition, focus |
| `utilities-interaction.md` | pointer |
| `utilities-color.md` | *(reserved — lands with the color-migration sprint)* |

Why:

- Component `llms.txt` is already 94KB (~23k tokens); inlining utilities would push past 130KB. An agent styling one screen needs 1–2 domains (~1.5–4k tokens each), not the whole surface — per-domain files cut context cost ~85–90% per task vs one file.
- Per-manifest files (20+) fragment related grammar (an agent doing layout wants margin + padding + sizes together) and bloat the index.
- The index makes discovery grep-free — exactly Mark's TOC idea, and it's what the llms.txt convention already means to agents: H1 + summary + `- [name](file.md): description` link lists.

## Phase 1 — manifest schema: `domain`

Add a required top-level `domain` field to `schemas/utility-manifest.schema.json`: enum `["layout", "spacing", "typography", "color", "border", "effects", "interaction"]`. Backfill all 20 manifests per the table above (ajv validation in the lint enforces it automatically). Manifest-level, not per-family — manifests are already domain-cohesive.

## Phase 2 — renderer

Extend `scripts/build-agent-context.ts` (split into `scripts/agent-context/{components,utilities,index}.ts` if the single file gets unwieldy — author's call). Load manifests via `MANIFEST_GLOB` and reuse `src/scripts/utilities/expand.ts` + `token-source.ts` (`resolvePatternValues`) so value vocabularies are derived, never re-typed. Follow the repo's `.js`-suffixed import style.

**Render grammar compactly — never enumerate expansions.** The manifests store the grammar; that's the whole token-economy win. Per family:

```markdown
### padding

Sets the empty space inside an element.

Grammar: `padding[-edge]-{value}` — edges: block, inline, block-start, block-end,
inline-start, inline-end · values: 3xs…3xl (spacing tokens), unset ·
responsive: prefix with `tablet:` `tablet-lg:` `desktop:` `desktop-lg:` `widescreen:`

- Zebkit uses logical padding operators (inline/block); 'block' is vertical, 'inline' is horizontal.
```

- Pattern families: the grammar line above (base, edges, value vocabulary with its token binding, negatives where present, responsive breakpoints) + `guidance` bullets + `a11y` line + `tier`/`valueTiers` (mark discouraged values inline) + user-visible aliases/`knownExceptions`.
- `classes` families: the class list with a one-line declarations summary (these are small).
- `rules` families: prose description + the classes they define (`classesFromRules`); the transition compound-modifier story and the `.prose` element styles get a short behavioral paragraph, not selector dumps.
- Each domain file opens with a shared preamble: the class grammar conventions (edge segments, responsive prefixes, negative values, `@layer utilities` overridability, tokens-not-hard-coded values) so family entries stay terse. Same `<!-- Generated by … -->` stamp as component docs.
- The renderer is mechanical — **do not invent guidance while rendering**. Editorial gaps get fixed in the manifests' `guidance`/`a11y` fields (that's the point of manifests-as-source-of-truth). Optional final pass: audit families with empty `guidance` and backfill the important ones for Mark's review.

## Phase 3 — index + full file

- Rewrite `llms.txt` to the llmstxt.org shape: H1, one-paragraph project summary (keep the existing grammar preamble, condensed), then sections **Components** and **Utilities** with `- [zbk-button](zbk-button.md): one-line purpose` / `- [Spacing utilities](utilities-spacing.md): margins, padding, sizes, page/section rhythm` lines. Relative links — the files sit side by side on disk in consumer repos.
- Component one-liners come from the manifest `purpose`; utility one-liners from a new short `summary` field or the manifest `description` (prefer `description`, add nothing new unless it reads badly).
- Generate `llms-full.txt`: the same concatenation llms.txt is today (preamble + every component doc) plus the utility domain files.
- `zebkit pull`/`init`: filter index lines for excluded components when copying `llms.txt` (line-per-doc makes this a cheap string filter — the "filtered llms.txt is the better product" item from the consolidation plan, now affordable). **Skip `llms-full.txt` in `copyAgentContext`** — it's a docs-site/hosted artifact; consumers get the individual files + index, and shipping it would leak excluded components' docs.

## Phase 4 — ship, tests, docs

- No packaging changes: `build:cli` copies the whole context dir; the `files` allowlist already covers it. Note the init/pull filename filter (`CONTEXT_COMPONENT_RE = /^zbk-(.+)\.md$/`) treats `utilities-*.md` as component-agnostic and always copies them — correct as-is.
- Tests beside the existing ones in `init.test.ts`/`pull.test.ts`: utilities files copied, `llms-full.txt` skipped, index lines filtered for excluded components.
- Docs: update the CLI page's description of what init/pull deliver (`docs/src/routes/cli/+page.mdx`); add a line to `src/scripts/utilities/README.md` ("manifests also feed the agent context docs — guidance edits regenerate them; `check` fails on drift").

## Verification

- `npm run check` green — `check:context` now drift-checks the utility files too; regenerate and stage together.
- Size audit: print per-file byte counts in the build log. Soft budget ≤ 15KB per domain file; if one blows past (grid is the likely one), split that domain rather than compressing the writing.
- Read-through QA: answer "how do I add responsive padding?", "which z-index token for a modal?", and "how do I make an element's background change on hover?" *(post-color)* from the relevant domain file alone, no grepping.

`<zbk-button>` wraps a real `<button>` in light DOM. The platform already ships the platonic button; the element completes the pattern — it renders the token-driven skeleton, reflects `variant` into classes, forwards native semantics, and adds the one behavior a native button lacks (`loading`). It carries zero visual opinion: every visual property is a `--zbk-button-*` token.

## Usage

```html
<zbk-button variant="ghost lg" type="submit">
  <svg slot="icon" viewBox="0 0 24 24">…</svg>
  Save draft
</zbk-button>
```

Renders (light DOM, inspectable):

```html
<zbk-button variant="ghost lg" type="submit">
  <button class="zbk-button zbk-button--ghost zbk-button--lg" type="submit">
    <span class="zbk-button__icon" aria-hidden="true"><svg>…</svg></span>
    <span class="zbk-button__label">Save draft</span>
  </button>
</zbk-button>
```

The rendered classes are the compilation target — stable and inspectable — so `.zbk-button` also works standalone on a native `<button>` when you need styling without JS. The element form is the documented entry point.

## Events

None custom. The native `click` bubbles out of the light DOM on its own — listen for `click` on `<zbk-button>` like any element. Keyboard activation (Enter/Space) arrives as a click, so one listener covers everything.

## Variants

A variant is a named, partial remapping of the token surface, compiled to a class. Anything not overridden falls through to the base token, so re-theming your aliases re-themes every variant automatically.

| Variant | Axis | Remaps |
|---|---|---|
| `ghost` | style | Transparent canvas + border; ink takes the action role. |
| `outline` | style | Transparent canvas, action-colored border. |
| `subtle` | style | Low-emphasis fill on the subtle action canvas. |
| `sm` | size | Smaller type/padding. Keeps the 44px tap-target floor. |
| `lg` | size | Larger type/padding. |

Different axes compose (`variant="outline lg"`); same-axis combinations warn in dev and the later class wins.

## Accessibility

- Label text is the accessible name; icon-only buttons must provide `aria-label` (a nameless button warns in dev with the fix).
- Write `aria-*`/`role` on `<zbk-button>` as if it were the button — they relocate to the inner `<button>` automatically.
- `loading` sets `aria-busy` and suppresses activation but keeps the button focusable, so keyboard and screen-reader users don't lose their place mid-operation.
- Focus styles ride `:focus-visible` with the shared focus tokens.

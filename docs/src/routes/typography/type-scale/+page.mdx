# Type scaling

Zebkit's font sizes are **generated, not hand-picked.** The entire scale — eleven named
steps from `3xs` to `5xl` — is derived at build time from six control values. Each step
fluidly interpolates between a small-screen size and a large-screen size, so type grows
with the viewport without a single media query, and the whole scale still answers to the
runtime accessibility dial.

<aside class="editorial-marginalia">
This is the premise from <a href="/foundations/tokens">the strata</a> taken to its
conclusion: the type scale isn't a list of values, it's a <em>function</em> of six
primitives. Redesigning it is a six-number diff.
</aside>

## Operate it

Drag the sliders. The first simulates a viewport width; the second is the end-user's
accessibility font-size modifier, multiplying through every size at once.

## Where it comes from

The method is [Utopia](https://utopia.fyi)'s fluid-type approach, rebuilt as a token
generator. Two ideas do the work:

- **Each step has its own slope.** Rather than adding one shared viewport increment to every
  size, every step interpolates between its own min and max. Big type gains more per pixel of
  viewport than body type does, so the hierarchy *fans out* on large screens instead of
  flattening.
- **The ratio itself is fluid.** The scale uses a tighter ratio at the small anchor (a minor
  third, `1.2`) and a more dramatic one at the large anchor (a major third, `1.25`). Adjacent
  steps sit exactly `1.2×` apart at 360px and exactly `1.25×` apart at 1240px.

## How it's built

Six control settings live in the `font-size` token module. They are the only literals in
the type system:

| Control | Default | Role |
| --- | --- | --- |
| `min-viewport` | `360px` | lower anchor |
| `max-viewport` | `1240px` | upper anchor |
| `min-base` | `1.125rem` | base (`md`) size at the lower anchor |
| `max-base` | `1.25rem` | base (`md`) size at the upper anchor |
| `min-ratio` | `1.2` | scale ratio at the lower anchor |
| `max-ratio` | `1.25` | scale ratio at the upper anchor |

Every named step carries an `index` (its position on the scale; `md` is `0`) instead of a
literal value. At build time the generator computes, for each step, its size at both anchors
(`base × ratioⁿ`), fits a line between them, and emits a `clamp()` — with the per-step
accessibility modifier baked into every term so the runtime dial keeps working:

```css
--zbk-font-size-md: clamp(
  calc(1.125rem * var(--zbk-a11y-font-size-modifier-md)),
  calc((1.0739rem + 0.2273vw) * var(--zbk-a11y-font-size-modifier-md)),
  calc(1.25rem  * var(--zbk-a11y-font-size-modifier-md))
);
```

These settings are build-time only — they're consumed during compilation and never emitted
as CSS variables. What ships is the eleven resolved `--zbk-font-size-*` properties.

Notice the two smallest steps <em>shrink</em> as the viewport grows — and the bounds are

## How to use it

You almost never touch the scale directly — you reference the named steps, and they stay
fluid automatically. The token names are the stable contract; the values float beneath them.

```css
.lede {
  font-size: var(--zbk-font-size-lg);
}
```

```html
<!-- Or via type utilities, which bind to the same tokens -->
<h1 class="text-3xl">Feature heading</h1>
<p class="text-md">Body copy that scales with the viewport.</p>
```

Because the sizes are tokens, every consumer — components, utilities, your own CSS — inherits
the fluid behavior and the accessibility dial for free. There is nothing to wire up.

## Tuning the scale

Re-skinning the type scale means changing the six controls, not the eleven steps. Override
them like any other token (in a theme override file or your custom token path):

```json
{
  "zbk-font-size": {
    "min-base": { "value": "1rem" },
    "max-base": { "value": "1.125rem" },
    "max-ratio": { "value": 1.333 }
  }
}
```

That single change re-derives all eleven steps in lockstep.

## Opting out: static sizes

If you want pixel-pinned control instead of fluid sizing, opt out in your zebkit config and
give the steps explicit values. Static sizes are still wrapped in the accessibility modifier,
so the runtime dial keeps working — they just don't respond to the viewport.

```json
{
  "tokens": {
    "typeScale": { "static": true }
  }
}
```

`fluid: false` is accepted as an alias for `static: true`. In static mode each step's `index`
is ignored and its authored `value` is used — so add a `value` to each step you want to pin:

```json
{
  "zbk-font-size": {
    "md": { "value": "1.125rem" }
  }
}
```

You can also pin a *single* step while the rest stay fluid: give just that step a `value` and
leave the config in its default fluid mode. The override wins for that step only.

## A quirk worth knowing

Because `min-ratio` is smaller than `max-ratio`, the scale fans out from the base in **both**
directions — which means steps below `md` invert: their small-viewport size is slightly
larger than their large-viewport size, so sub-body text shrinks a touch as the screen grows
(`3xs` goes 8.7px → 8.2px). The clamp bounds are ordered by magnitude so this interpolates
smoothly rather than locking up. The deltas are sub-pixel, and a reader who needs larger text
has the accessibility dial — so we let it stand as designed. If you'd rather small text never
shrink, pin those steps with static values as shown above.
